# Blender Python API — 5.0 → 5.1 deltas

Source: https://docs.blender.org/api/5.1/change_log.html (section "5.0 to 5.1").
Filtered to the categories that affect procedural mesh building, materials,
EEVEE rendering, and URDF/GLB export — i.e. what Topos actually uses.

## TL;DR for Topos

- **Most procedural mesh code keeps working unchanged.** `bpy.ops.mesh.primitive_*`, `bmesh.ops.*`, `obj.modifiers.new`, transform helpers — all unchanged in surface.
- **EEVEE got new top-level intensity controls** (`SceneEEVEE.direct_light_intensity`, `SceneEEVEE.indirect_light_intensity`). The values we currently set on individual light data still work; the new scene-level multipliers are additive sliders we don't have to touch.
- **Compositor was rewritten** — ~80 `bpy.types.CompositorNode*` subclasses were removed and consolidated. Topos does not use the compositor at all, so this is a non-issue for us.

## Changes that matter to Topos code paths

### EEVEE rendering (`render_wrapper.py` territory)

| Type | Change |
|---|---|
| `bpy.types.SceneEEVEE` | **Added:** `direct_light_intensity` (default 1.0), `indirect_light_intensity` (default 1.0) — scene-level multipliers on top of per-light energy values. Backwards-compatible: omit them and rendering looks the same as 5.0. |

### Shading (`material_attach.py` and any agent that writes shader nodes)

| Type | Change |
|---|---|
| `bpy.types.ShaderNodeNormalMap` | **Added:** `base`, `convention` — new properties on the normal map node. Existing default-constructed normal maps unaffected. |

### Geometry (procedural primitives — what part agents emit)

| Type | Change |
|---|---|
| `bpy.types.Curve` | **Added:** `fill_rule`, `fill_solver` — new curve fill controls. We don't currently use curves; relevant only if a future part is curve-based. |
| `bpy.types.GeometryNodeStringToCurves` | **Removed:** `align_x`, `align_y`, `font`, `overflow`, `pivot_mode` — only relevant if you use Geometry-Nodes-based text generation. |
| `bpy.types.Object` | **Added:** `shape_keys_selected` — only relevant for shape-key animation; unused here. |

### Compositor — entire subsystem rewritten

Skip unless you touch `bpy.context.scene.use_nodes` for the compositor (not
the shader node tree). Removed classes include `CompositorNodeBlur`,
`CompositorNodeBokehBlur`, `CompositorNodeChromaMatte`,
`CompositorNodeCryptomatte`, and ~75 others. The 5.1 compositor uses a
single unified node API. **Topos: not used → no migration needed.**

## What to do when upgrading the pinned Blender to 5.1

1. Run `topos bpy-docs index` to rebuild the index against the new binary.
2. Rerun the cabinet example — should pass with the same or better score (no API call changes required).
3. Optional: try the new `SceneEEVEE.direct_light_intensity` to dial down the bright lighting we currently get with EEVEE-default. Set it to 0.8–1.0 in `render_wrapper.py` if you want softer baseline lighting.

## Things to watch (not changes — pre-existing patterns that still apply)

- `bpy.ops.mesh.primitive_cube_add(size=2)` — default size is 2 (2m cube, ±1m corners), not 1. Easy off-by-2 mistake. Unchanged from 5.0.
- Active object context matters for `bpy.ops.*` calls. Always `bpy.context.view_layer.objects.active = obj` before applying modifiers via ops. Unchanged from 5.0.
